Gluten + Velox Architecture

How It Works

Architecture at a Glance

Apache Spark — Distributed Control Flow, Scheduling, APIs, Fault Tolerance
▼ Spark Physical Plan
Apache Gluten — Plan Translation (Substrait IR) → JNI Bridge
Spark Plugin API — no code changes required
▼ Native Execution
Velox — Vectorized C++ Engine (Meta)
Columnar batches · SIMD · Off-heap memory · Zero GC
Data Format: Apache Arrow ColumnarBatch · Fallback: ColumnarToRow / RowToColumnar

Why This Design Wins

Design PrincipleBenefit
Reuse Spark's Control PlaneKeep scheduling, shuffle service, fault tolerance, APIs — no migration
Offload Compute to NativeVectorized C++ with SIMD — 2–6× faster on same hardware
Substrait as IRCross-engine plan representation — portable, extensible
Graceful FallbackUnsupported ops run on JVM Spark — no query failures
Plugin APIZero code changes — just configure spark.plugins
Key insight: Gluten doesn't replace Spark — it makes Spark's hot path native while preserving everything else.

Production Experience

Challenges at Scale

What Worked Well

✅ Drop-In Acceleration

Existing Spark jobs ran faster with spark.native.enabled=true — no rewrites needed.

✅ Broad Operator Coverage

Scans, filters, projections, aggregations, joins, sorts — the hot path was native from day one.

✅ Significant Speedups

Up to 6× end-to-end on TPC-DS SF1000. Compute-intensive queries saw the biggest gains.

✅ Graceful Fallback

Unsupported operators fall back to JVM Spark transparently — no query failures.

Challenges We Faced

ChallengeWhat HappenedHow We Addressed It
Correctness Subtle differences in null handling, cast semantics, and edge cases between Spark and Velox Extensive testing + upstream fixes (e.g., assert_not_null, collect_set null behavior)
Fallback Behavior Excessive ColumnarToRow/RowToColumnar transitions degraded performance Fallback advisor in Spark UI + dynamic filter pushdown to reduce transitions
Memory Management JVM on-heap vs. Velox off-heap competition caused OOM Dynamic memory negotiation — OOM issues dropped 70% after Gluten 1.3.0
Security Integration Native code paths needed to honor Fabric's security model JNI bridge security validation + sandboxed native execution
Operational Stability Native crashes harder to diagnose than JVM exceptions Enhanced diagnostics, query trace, Spark Advisor fallback alerts

Workload Patterns That Benefit Most

🟢 Great for Native Execution

  • Complex aggregations — GROUP BY with many expressions
  • Multi-way joins — hash joins, broadcast joins
  • Heavy filtering & projections — columnar + SIMD shines
  • Parquet / Delta reads — native scan with pushdown
  • Compute-intensive ETL — CPU-bound transformations

⚠️ Consider Carefully

  • I/O-bound scans — limited compute to accelerate
  • UDF-heavy workloads — Python/Java UDFs run on JVM
  • Unsupported ops — frequent fallbacks add overhead
  • Small datasets — overhead may exceed savings
  • Streaming workloads — support still maturing

Fabric Native Execution Engine

Differentiation at Microsoft

What Is NEE?

The Native Execution Engine (NEE) is Microsoft Fabric's integration of Gluten + Velox — available to all Fabric Spark users at no additional cost.

Fabric Spark Runtime 1.3 — Spark 3.5 + Delta Lake 3.2 + Fabric Extensions
▼
Native Execution Engine — Gluten + Velox (C++ Vectorized)
▼
OneLake — Unified Lakehouse Storage (Delta / Parquet)
spark.native.enabled = true

NEE Performance Results

~4× Faster on TPC-DS 1TB
Up to 6× End-to-End on Production Jobs
~83% Compute Cost Savings
$0 Additional Cost
TPC-DS SF1000 (Delta format): Up to 6× faster vs. open-source Spark on a fixed Fabric cluster — translating to ~83% compute-cost savings. No extra charge — same Spark credit rates apply.

How NEE Differs from OSS Gluten

Open-Source Gluten

  • Manual build & deployment
  • User manages Velox libs
  • Community support
  • DIY configuration tuning
  • Standard Spark monitoring

Fabric NEE

  • Pre-integrated in Fabric Runtime
  • One toggle: spark.native.enabled
  • Enterprise support & SLAs
  • Fabric-tuned defaults for Delta/OneLake
  • Spark Advisor — real-time fallback alerts

Fabric-Specific Optimizations

OptimizationDescription
V-OrderDefault columnar sorting for Parquet/Delta — optimized read patterns for analytical queries
OneLake IntegrationUnified lakehouse storage with optimized I/O paths for native scans
Native Delta WriteAccelerated Delta write path — bypassing JVM serialization
Deletion VectorsNative support for Delta deletion vectors — efficient row-level deletes
Optimized SnapshotsDriver-side Delta snapshot management — faster metadata operations
Dynamic Memory ManagerRuntime 1.3 — smart JVM/native memory negotiation — fewer OOMs
Z-Order / Liquid ClusteringNEE-aware data layout optimizations for faster scans
These Fabric-specific enhancements compose with upstream Gluten/Velox improvements — the best of both worlds.

Observability & Developer Experience

Lessons from the Journey

🏛️ Governance > Code

ASF requires diverse PMC. Early on, 90% of contributions came from one company. Graduation required deliberate diversification.

🧠 Memory Is the Hardest Problem

Spark on-heap vs. Velox off-heap competed for RAM. Dynamic negotiation took 9 months — but dropped OOM issues by 70%.

🔄 Production Feedback Is Gold

Bugs found at Fabric scale — null handling, cast semantics, edge cases — directly shaped upstream design decisions.

🤝 Multi-Company = Resilience

Microsoft, Meta, Intel, Kyligence, BIGO, Meituan, Alibaba — diverse contributors make the project stronger.
